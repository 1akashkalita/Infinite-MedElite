<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/export/pdf | The view-model arrives as a client-controlled JSON POST body; it is untrusted until `ReportViewModelSchema.safeParse` succeeds. |
| validated model → PDF renderer | Only `parseResult.data` (post-safeParse) reaches `renderToBuffer`; the renderer never sees raw input. |
| validated model → Content-Disposition header | `displayName` and `ccn` flow into the response filename; raw bytes could carry header-injection/path-traversal payloads if not sanitized. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | POST body (client-controlled ReportViewModel) | mitigate | KEEP the existing `ReportViewModelSchema.safeParse(body)` gate; only `parseResult.data` reaches `renderToBuffer`. No unvalidated data reaches the renderer (V5). |
| T-04-02 | Injection | `careCompareUrl` rendered into a PDF `<Link>` annotation | mitigate | The schema origin-pins `careCompareUrl` to `https://www.medicare.gov` (protocol + hostname refine in view-model.ts). The `<Link>` consumes the validated value as-is; no `javascript:`/`data:` URI can reach the annotation. |
| T-04-03 | Tampering | `Content-Disposition` filename derived from `displayName` | mitigate | `slugFilename` strips every non-`[a-z0-9]` char to hyphens before composing the filename, so quotes/CRLF/slashes/`..` cannot survive into the header — tested as an acceptance criterion (injection-safety case in slug.test.ts). |
| T-04-04 | Information Disclosure | 400 error responses on bad/invalid bodies | accept (already mitigated) | The pre-existing clean-envelope discipline (no Zod internals — `{ error: { kind, message } }`) is preserved unchanged; the route swap touches only the success branch. |
| T-04-SC | Tampering | npm/pip/cargo installs | n/a | No new packages installed this phase — `@react-pdf/renderer` and `zod` passed the Phase 1 legitimacy checkpoint (04-RESEARCH.md Package Legitimacy Audit). No install task exists. |

No `high`-severity threats identified. The PDF route adds no new untrusted-input surface beyond the already-validated POST body; the only new sink (the filename header) is sanitized and tested.
